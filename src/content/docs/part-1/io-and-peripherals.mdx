Every device on the HEX-816 that is not memory is reached through one 256-byte
window: the I/O page at `$CF00`–`$CFFF` in bank&nbsp;0. This chapter is the map of
that window — which **SYS** GPIO does what, how an I/O access is served, and what
sits behind each block of registers. The register addresses themselves are
listed in full in Appendix&nbsp;A; here the focus is the wiring and the behaviour.

## SYS on the CPU bus

**SYS** (one of the three RP2354B coprocessors) is the only chip that touches the
65C816 directly. It generates the clock, samples every bus cycle, drives the
control lines, and decodes the I/O page. The other two coprocessors — **VID**
(PRISM video) and **SND** (sound) — never see the CPU bus; they are reached
through SYS over **SPI** and **QSPI**.

The 65C816 multiplexes its bus: during the low half of each `PHI2` cycle the data
pins carry the high address bits `A16–A23`, and during the high half they carry
read or write data (Chapter&nbsp;2). SYS reads the low address bits directly,
latches the bank byte, and recovers the full 24-bit address in firmware.

### GPIO assignment

SYS is an RP2354B in a QFN-80 package; all 48 of its GPIOs are allocated. The
complete map:

| GPIO | Net | Dir | Role |
| --- | --- | --- | --- |
| `GP0`–`GP15` | `A0`–`A15` | in | Low 16 address bits, straight from the CPU |
| `GP16`–`GP23` | `D0`–`D7` | bidir | Data bus — bank byte in `PHI2` low, data in `PHI2` high |
| `GP24` | `PHI2` | out | CPU clock, generated by a PIO state machine |
| `GP25` | `RWB` | in | Read/write strobe |
| `GP26` | `VDA` | in | Valid data address |
| `GP27` | `VPA` | in | Valid program address |
| `GP28` | `RDY` | out | Wait-state / stall line |
| `GP29` | `RESB` | out | CPU reset |
| `GP30` | `IRQB` | out | Aggregated maskable interrupt |
| `GP31` | `NMIB` | out | Non-maskable interrupt (reserved; see below) |
| `GP32` | `RTC_CE` | out | RTC chip-enable — bodged to the CPU `BE` line for boot preload |
| `GP33` | `Y7_FB` | in | `74AHC138` `Y7` feedback (high-bank hint) |
| `GP34` | `CS_VID_SPI` | out | VID SPI slave-select (firmware-update path) |
| `GP35` | `SD_CS` | out | SD card SPI chip-select |
| `GP36` | `RUN_VID_CTRL` | out | VID run/reset control |
| `GP37` | `QSPI_SCK` | out | QSPI clock to VID |
| `GP38`–`GP41` | `QSPI_D0`–`D3` | out | QSPI data to VID (PRISM command stream) |
| `GP42` | `SPI_SCK` | out | SPI clock (SND, and VID extended) |
| `GP43` | `SPI_MOSI` | out | SPI data out |
| `GP44` | `SPI_MISO` | in | SPI data in |
| `GP45` | `SPI_CS` | out | SND chip-select |
| `GP46` | `RTC_IO` | bidir | DS1302 serial data |
| `GP47` | `RTC_SCLK` | out | DS1302 serial clock |

:::note[Note · bring-up]
`GP32` is the RTC chip-enable in the production design, but on Rev&nbsp;2.1 it is
bodged to the CPU's `BE` (bus-enable) line so SYS can hold the CPU off the bus
while it preloads the BIOS at boot. The RTC chip-enable is sacrificed for that
window. Rev&nbsp;3.0 routes `BE` to its own pin and returns `GP32` to the RTC.
:::

### Serving an I/O access

When the CPU reads or writes an address in `$00:CF00`–`$00:CFFF`, SYS recognises
it, strips the `$CF00` base to get an 8-bit page offset, and dispatches on that
offset. Internal registers (the 60&nbsp;Hz tick, the keyboard byte, the reset and
debug registers) are handled on SYS itself; the rest are forwarded to a
peripheral over SPI or QSPI. Offsets that decode to nothing read `$FF` and ignore
writes.

Because the `74AHC138` decoder selects the SRAM on *every* bank-0 access — `A0–A15`
are not part of its decode — a `$CFxx` access also selects the SRAM. The current
firmware lives with that; suppressing the SRAM for the I/O page is the subject of
the forward-looking serve redesign and is not active on Rev&nbsp;2.1.

## Console and USB

The console UART is the operating system's terminal. HEX/OS reads and writes
`UART_DATA` at `$CF10` and polls `UART_STATUS` at `$CF11`; SYS relays those bytes
over SPI to **SND**, which exposes them as a USB CDC ACM serial port (CDC1) to a
host PC. A one-byte SPI discriminator distinguishes traffic on the shared link —
`'C'` (`$43`) tags a console transmit byte and `'c'` (`$63`) a console receive
poll — so the console shares the SND SPI channel with the audio registers and the
debug-adapter port without collision (Chapter&nbsp;5 covers the SND side).

SYS is also the USB **host**. Through an **FE1.1S** hub it enumerates a USB
keyboard, whose keystrokes appear at `KBD_DATA` (`$CF05`) with a ready flag at
`KBD_STATUS` (`$CF06`), and the **ESP32-C3** networking module (below). A pressed
key also raises `IRQB` so the OS need not poll.

## Storage — SD card and FAT

The SD card is an SPI device on `GP42`–`GP45` with its chip-select on `GP35`. SYS
runs a **FatFs** filesystem on top of it and exposes a register-based command
interface rather than raw blocks. Writing a command code (`≥ $10`) to `SD_CMD`
(`$CF20`) dispatches a filesystem operation — open, read, write, seek, status,
directory search, rename, delete, mkdir, sync — and the `$CF28`–`$CF39` register
file carries the arguments and results (handle, path string, position, attributes,
size, name, and date). Bulk data moves 128 bytes at a time through the `SD_DATA`
stream port at `$CF25`, the CP/M record size. Storage is covered in depth in
Chapter&nbsp;3.

:::caution
On the current firmware, writes to `SD_DATA` (`$CF25`) are not routed to the
filesystem handler, so the `WRITE128` save path does not deliver its bytes — SD
*reads* work but *saves* do not yet. `SD_STATUS` (`$CF26`) is likewise
unimplemented and reads `$FF`. Both are tracked as firmware decode issues
(Appendix&nbsp;A).
:::

## Real-time clock

A **DS1302** keeps wall-clock time on three GPIOs — chip-enable (`GP32`), bidirectional
data (`GP46`), and clock (`GP47`). SYS maintains a RAM shadow of the seven
timekeeping registers, refreshed once a second, so CPU reads of `RTC_SEC` through
`RTC_YEAR` at `$CF50`–`$CF56` return instantly without waiting on a bit-banged
transaction. Values are BCD. Writes stage into the shadow and commit to the chip
when bit&nbsp;0 of `RTC_CTRL` (`$CF57`) is set.

## Networking

Wi-Fi and TCP/IP run on an **ESP32-C3** module, reached not over the CPU bus but
as a USB CDC device on the SYS hub. SYS frames command/response packets to it and
presents a register interface at `$CF58`–`$CF67`: a status byte, a command
register, a result code, a socket handle, a string buffer for SSIDs and hostnames,
a data FIFO, transfer-length and port registers, the resolved IP address, and the
Wi-Fi signal strength. The command set covers scanning and connecting to an access
point, DNS resolution, TCP connect/send/receive/close, an HTTP GET, NTP time sync,
and ping.

## Audio and video

**SND** audio registers are written through the `$CFA0`–`$CFBF` window: SYS
forwards each write to SND over SPI as a two-byte packet of `{offset, value}`.
This drives the 16-voice PSG and the four-channel PCM mixer (Chapter&nbsp;5).

**VID** receives the PRISM command stream through a single port, `PDI` at `$CFE0`
— each write sends one byte to VID over the QSPI link (`GP37`–`GP41`). VID runs the
command list on its own core and shifts HDMI out directly; there is no separate
display controller to program (Chapters&nbsp;4 and&nbsp;11).

:::caution
The `$CFA0`–`$CFBF` window only reaches SPI registers `$00`–`$1F` on SND, which is
narrower than the full audio register range, and the `PDI` port at `$CFE0`
overlaps where the PCM sample-pool registers would extend. Neither has been
reconciled yet (Appendix&nbsp;A).
:::

## Debug

The `$CF70`–`$CF72` registers carry the BRK debug handshake. When the CPU executes
`BRK`, the BIOS handler saves the registers and writes the magic byte `$CC` to
`$CF70`; SYS notices, snapshots the machine state, and the host debug adapter
drives single-step, continue, or reset back through `$CF71`. Chapter&nbsp;15 walks
through the full protocol.

## Interrupts

SYS owns both CPU interrupt lines. Every peripheral source — the 60&nbsp;Hz timer
tick at `$CF03`–`$CF04`, keyboard input, SD-card-ready, an incoming network packet
— is aggregated by SYS and delivered to the CPU as a single `IRQB` assertion on
`GP30`; the interrupt handler then queries SYS to learn which source fired. The
`NMI` line (`GP31`) is reserved for the PRISM VSYNC interrupt; on Rev&nbsp;2.1 it
is not wired, and the raster effects that depend on it are described in
Chapter&nbsp;11.
